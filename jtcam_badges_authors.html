FAIR Software

🌐 Accessibility

🌐 Level 1

code available on a public forge

The source code can be found on a git force host gitrepositories. For instance:

  • GitHub — Widely used; strong integration tools; limited openness (proprietary).
  • GitLab — Open-core; self-host or cloud; good CI/CD; FAIR-aligned transparency.
  • Codeberg — Fully open-source (Gitea-based); privacy-friendly; EU-hosted.
  • SourceHut — Minimalist; open protocols; plain-text workflows; fully libre.
  • Bitbucket — Proprietary; integrates with Atlassian suite; less open.
  • Gitea — Lightweight, self-hosted, MIT-licensed; supports FAIR openness.
  • Forgejo — Gitea fork; community-driven; strong open governance.
  • GRICAD GitLab — Academic research-forge (Université Grenoble Alpes + CNRS + INRIA); local GitLab instance for research software. :contentReferenceoaicite:0
  • IRD Forge — Institutional GitLab-based forge by Institut de recherche pour le développement (IRD); supports versioning, CI/CD, public/private code. :contentReferenceoaicite:2
  • Forgejo / Fordj (France-specific interest) — Libre community-forge (via DINUM / BlueHats) for French public sector; open-source development. :contentReferenceoaicite:4
Software commit/branch used for publication results
  • Within the README file: Doe, J., & Smith, A. (2025). CoolSim: Climate simulation toolkit (Version 2.3.1) [Computer software]. GitHub. Commit: 4f8a9d2. https://github.com/exampleuser/coolsim/tree/4f8a9d2

Open-source License

The License must be clearly indicated in the root of the repository (LICENSE/COPYING). Exemple of valid licenses:

  • MIT License — Permissive; free reuse, modify, distribute; attribution only.
  • Apache 2.0 — Permissive; attribution + patent grant; high interoperability.
  • BSD 3-Clause — Permissive; attribution + non-endorsement.
  • GPL-3.0 — Copyleft; derivatives must remain open.
  • MPL-2.0 — Weak copyleft; file-level openness; flexible reuse.
  • CC-BY-4.0 — Attribution; for data/docs; FAIR-compliant openness.
Authors and contributors provided

The authors and contributors should be given with one or several of the following:

  • AUTHORS file — Plain text or Markdown file listing main authors and maintainers.
    Example: AUTHORS, AUTHORS.md

  • CONTRIBUTORS file — Lists additional contributors or institutional credits.
    Example: CONTRIBUTORS, CONTRIBUTORS.md

  • Within README file — Optional “Authors” or “Credits” section; human-readable summary.

  • Within Version control management — With clean commit history, the exact contributor list is metadata.

🌐🌐 Level 2

Code archived on Zenodo
  • Code archived on Zenodo - Repository release exported to Zenodo for citation and FAIR compliance. Long-term preservation and DOI assignment.
  • Can be done via GitHub–Zenodo integration.

🌐🌐🌐 Level 3

Software Heritage Identifier (SWHID)
  • Code archived on Software Heritage — Ensures permanent, tamper-proof preservation of source code. Every commit is archived and globally accessible.
  • Very easy and reliable.
  • Provide a SWHID permanent id.
  • See Software Heritage: Archiving GitHub repositories for instructions on submitting repositories.

🌐🌐🌐🌐 Level 4

HAL deposit
  • Code deposited in HAL — Upload your software as a research output to HAL.
  • Provides an open-access archive with DOI assignment.
  • Supports detailed metadata, linking to authors, grants, and publications.
  • Long-term preservation with institutional endorsement.

Resources (summary)

ACM badges 1/2

One badge by element. Can be attributed independently and cumulated.
“artifact” = digital object created by the authors to be used as part of the study or generated by the experiment itself.

Badge Validated elements
Artifacts evaluated: functional (available for the reviewer) Documented, consistent, complete, exercisable, includes verification/validation
Artifacts evaluated: re-usable (available for the reviewer) Exceeds minimal functionality, carefully documented, well-structured, norms and standards adhered

ACM badges 2/2

Badge Description
Artifacts available (publicly) Author-created artifacts placed on a public repository with DOI or unique identifier
Results reproduced Main results obtained by another team using artifacts provided by authors
Results replicated Main results independently obtained without author-supplied artifacts

Source: Replicability Stamp. Papers with Replicability Stamp cited 15% more after 1 year, 33% more after 2 years (WoS 2020).

Benureau / Rougier

Level State Definition
R¹ re-runnable Code can be run again as needed
R² repeatable Produces same output over successive runs
R³ reproducible Another researcher can re-obtain same result with original code/data
R⁴ re-usable Easily used and modified by others (comments, documentation, workflow)
R⁵ replicable Article description allows results to be re-obtained without source code

CODE beyond FAIR 1/2

OPEN

Recommendation Level
Publish source code on public forge mandatory
Save repository on dedicated archive mandatory
License code with an open license strongly recommended
Declare authorship and rightholders recommended

DOCUMENT

Recommendation Level
Choose meaningful names recommended
Comment code recommended
Provide examples, notebooks, or tutorials recommended
Describe API optional

CODE beyond FAIR 2/2

EXECUTE

Recommendation Level
List software and hardware dependencies recommended
Provide computational environment optional
Implement a test suite optional
Show real-life usage example with expected results optional

COLLABORATE

Recommendation Level
Respond to issues and feature requests optional
Describe maintenance, features, support limits optional
Explain how to contribute optional
Build and animate a community optional
