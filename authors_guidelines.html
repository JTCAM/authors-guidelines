Computational Paper Guidelines

FAIR Data

Level 1

  • Data available in an accessible permanent repository (DOI)
  • Free license (e.g., Creative Commons)
  • File listing authors and contributors

Level 2

  • Open file formats (e.g., HDF5, NetCDF)
  • Complete metadata file
  • Documentation

Level 3

  • Curation and validation ensuring consistency between content and paper

FAIR Software

Level 1

NoteCode hosted on a forge with version, commit number or Zenodo DOI

git forges:

  • GitLab — Open-core; self-host or cloud; good CI/CD; FAIR-aligned transparency.
  • GitHub — Widely used; strong integration tools; limited openness (proprietary).
  • Codeberg — Fully open-source (Gitea-based); privacy-friendly; EU-hosted.
  • SourceHut — Minimalist; open protocols; plain-text workflows; fully libre.
  • Bitbucket — Proprietary; integrates with Atlassian suite; less open.
  • Gitea — Lightweight, self-hosted, MIT-licensed; supports FAIR openness.
  • Forgejo — Gitea fork; community-driven; strong open governance.
  • GRICAD GitLab — Academic research-forge (Université Grenoble Alpes + CNRS + INRIA); local GitLab instance for research software. - IRD Forge — Institutional GitLab-based forge by Institut de recherche pour le développement (IRD); supports versioning, CI/CD, public/private code. :contentReferenceoaicite:2
  • Forgejo / Fordj (France-specific interest) — Libre community-forge (via DINUM / BlueHats) for French public sector; open-source development. :contentReferenceoaicite:4

NoteFree license

The License must be clearly indicated in the root of the repository (LICENSE/COPYING). Exemple of valid licenses:

  • MIT License — Permissive; free reuse, modify, distribute; attribution only.
  • Apache 2.0 — Permissive; attribution + patent grant; high interoperability.
  • BSD 3-Clause — Permissive; attribution + non-endorsement.
  • GPL-3.0 — Copyleft; derivatives must remain open.
  • LGPL-3.0 — Copyleft; Lesser GPL permits use of the library in proprietary programs.
  • MPL-2.0 — Weak copyleft; file-level openness; flexible reuse.
  • CC-BY-4.0 — Attribution; for data/docs; FAIR-compliant openness.
NoteAuthors and contributors file

The authors and contributors should be given with one or several of the following:

  • AUTHORS file — Plain text or Markdown file listing main authors and maintainers. Example: AUTHORS, AUTHORS.md

  • CONTRIBUTORS file — Lists additional contributors or institutional credits. Example: CONTRIBUTORS, CONTRIBUTORS.md

  • Within README file — Optional “Authors” or “Credits” section; human-readable summary.

  • Within Version control management — With clean commit history, the exact contributor list is metadata.

Level 2

  • Documented code (inline + user documentation)
  • “How to use it?” instructions (README, scripts, etc.)
  • Metadata file

Level 3

  • Examples
  • Automated tests
  • Functionality documentation (e.g., API documentation)

Reproducible Results

Level 1

  • Persistent identifier (commit, SWHID, DOI, etc.)
  • Reproducibility recipe (code execution + post-processing)
  • README and scripts

Level 2

  • Full hardware/software environment (Guix, venv, requirements.txt, renv, etc.)
  • Workflow manager for reproducibility
